==================================================
1) PROJECT OVERVIEW
==================================================
TruthESG is a React + FastAPI application for ESG report checking.
Core user journey:
1. User uploads a PDF ESG report (or searches a company name for a mock PDF).
2. Backend extracts text from PDF.
3. Backend identifies ESG-like claims.
4. Backend scores each claim as Verified / Questionable / High Risk.
5. Frontend renders verdict cards, scores, and reasons.

Current production pipeline is heuristic/rule-based.
There is a separate ai/ module containing agent-like logic that is not integrated into backend routes yet.

A) GET /health

    "status": "ok"

B) POST /upload-pdf
- File: backend/app/routes/upload.py
- Handler: upload_pdf(file: UploadFile)
- Content-Type expected: multipart/form-data
- Required field: file (UploadFile)
- Validation:
  - Rejects when file.content_type != "application/pdf".
  - Returns HTTP 400 with detail "Only PDF files are allowed".
- Processing:
  - Calls store_pdf(...) in backend/app/services/pdf_service.py.
  - store_pdf calls save_uploaded_file(...) in backend/app/utils/file_handler.py.
  - Saves to backend/uploads/{uuid}_{original_filename}.
- Response model: UploadPDFResponse
  {
    "file_path": "<absolute path to saved file>"
  }
- Status codes:
  - 200 success
  - 400 invalid MIME type

C) POST /analyze
- File: backend/app/routes/analysis.py
- Handler: analyze(file: UploadFile)
- Content-Type expected: multipart/form-data
- Required field: file (UploadFile)
- Validation:
  - Rejects non-PDF MIME type with HTTP 400.
- Pipeline:
  1) Save uploaded PDF to backend/uploads via store_pdf(...)
  2) Extract and process claims via process_pdf(file_path)
     - extract_text_from_pdf(...) using PyMuPDF
     - split_text(...) into ~2000-char chunks with boundary heuristics
     - extract_claims_from_chunk(...) with keyword/percent matching
     - deduplicate claims
  3) Score each claim via score_claim(claim)
  4) Return normalized list of FinalClaimItem
- Response model: AnalyzeClaimsResponse
  {
    "claims": [
      {
        "claim": "...",
        "final_score": 0.75,
        "final_risk": "Verified",
        "reason": "Strong measurable claim with specific data"
      }
    ]
  }
- Error mapping:
  - FileNotFoundError -> HTTP 404
  - ValueError -> HTTP 400
  - RuntimeError -> HTTP 500

D) GET /
- File: backend/app/main.py
- Handler: root()
- Purpose: base status/info endpoint.
- Response:
  {
    "message": "TruthESG Backend is running",
    "docs": "/docs"
  }
- Status code: 200.

E) GET /favicon.ico (not in OpenAPI schema)
- File: backend/app/main.py
- Handler: favicon()
- Purpose: avoid favicon 404 noise.
- Response: empty body, status 204.

4) SCORING + CLAIM EXTRACTION LOGIC
==================================================
Claim Extraction (backend/app/services/pdf_service.py)
- extract_text_from_pdf(file_path)
  - Uses fitz.open(...) from PyMuPDF.
  - Concatenates page text in order.
  - Throws:
    - FileNotFoundError if file missing
    - ValueError for empty PDF or no extractable text
    - RuntimeError for extraction failure

- split_text(text, chunk_size=2000)
  - Splits large text into chunks.
  - Tries to split near newline or sentence punctuation boundaries.

- extract_claims_from_chunk(chunk)
  - Splits into sentences.
  - Keeps sentence if either:
    - contains "%"
    - OR contains one of: reduced, increased, achieved, improved, decreased
  - Deduplicates case-insensitively.

- process_pdf(file_path)
  - Full orchestrator: extract -> split -> claims -> dedupe -> return list[str].

Scoring (backend/app/services/scoring_service.py)
- Baseline score = 0.5
- Positive signals:
  - +0.1 if claim contains "%"
  - +0.1 if contains any number
  - +0.1 if contains action word (reduced/decreased/achieved/implemented)
  - +0.15 if contains verification word (audited/certified/verified)
- Negative signals:
  - -0.15 if contains intent words (aim/target/plan/commit)
  - -0.1 if contains marketing words (leading/best/sustainable/eco-friendly)
  - -0.1 if contains "will" or regex "by 2030"
- Final clamp: [0.0, 1.0], rounded to 2 decimals.
- Risk label mapping:
  - >= 0.75 => Verified
  - >= 0.45 => Questionable
  - < 0.45 => High Risk
- Reason text generated from risk bucket.

Note: calculate_cps(...) also exists in this file but is not used by current routes.

5) FRONTEND FLOW DETAILS
==================================================
Main orchestration: frontend/src/App.tsx

Direct upload flow
1. User picks PDF in UploadCard.
2. onFileSelect(file) accepts only application/pdf.
3. handleAnalyze() creates FormData and POSTs to http://localhost:8000/analyze.
4. While processing, AnalysisLoadingOverlay is displayed.
5. Response claims are shown in dashboard cards.

Company search flow
1. CompanySearchBar collects text query.
2. getCompanyPDF(query) maps names to mock PDF path.
3. App fetches that PDF as blob.
4. Blob is submitted to /analyze exactly like uploaded file.

Derived UI analytics in App.tsx
- avgScore = mean(final_score across claims)
- overallRisk thresholds mirror backend buckets.
- confidencePercent = round((1 - avgScore) * 100)
- verdict and key insight strings depend on avgScore and questionable-count.

8) KNOWN GAPS / IMPORTANT NOTES FOR AN AI AGENT
==================================================
1. AI module is not integrated into active API routes.
- backend analyze path uses backend/app/services/scoring_service.py only.
- ai/agents and ai/claim_extraction are currently standalone.

2. Schema drift / unused models exist.
- In schemas.py, AnalyzeRequest, AnalyzeResponse, ScoredClaimItem are currently unused by active endpoints.

3. Company mock-data mismatch.
- getCompanyPDF.js can return reliance/adani/tata paths.
- Only Tata.pdf exists in frontend/public/mock-data right now.
- Searching Reliance/Adani will likely fail with file-not-found unless those PDFs are added.

4. Frontend backend URL hardcoded.
- App.tsx uses http://localhost:8000 directly, no env abstraction.

5. CORS is fully open by default.
- settings.py allows all origins/methods/headers (fine for local dev, risky for prod).
